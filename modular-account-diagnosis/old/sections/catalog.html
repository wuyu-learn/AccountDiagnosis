卡片取值方式目录（815 上线版）
本目录说明 17 张账户卡片的数据取值方式（dataMode）与召回交互，供前端开发确定每张卡"如何取数、何时显示"。核心结论：每个卡片有固定的 dataMode，由卡自身性质决定，不因"被怎么打开"而改变；卡片召回分两种入口（直接问 / 点击下钻），入口只决定何时触发，不改取数方式。

一、取值方式定义（dataMode）

每个卡片有固定的 dataMode，由卡片自身性质决定，不因"被怎么打开"而改变：

模式含义适用
inlineOpenClaw 直接输出完整 data，前端直接渲染、不二次取数。数据量小、无复杂交互、无长时序走势的汇总 / 列表卡。
deferredOpenClaw 仅输出 dataMode:'deferred' + data.request（标题 + 参数），前端再调专用 API 取数后渲染；取数失败则整卡隐藏。数据量大、多 tab / 维度切换、长时序走势图、需穿透计算的卡。

二、召回交互（两种入口，只决定"何时触发显示"）

召回入口说明
① 直接问（query）用户在对话中直接提问，OpenClaw 召回该卡。dataMode 不变。
② 点击进入（下钻）从另一张卡片的按钮 / 格点击进入（见第三节下钻关系网）。dataMode 不变。
关键原则：入口（直问 or 点击）只影响"这张卡何时被触发"，不影响取数方式。前端统一逻辑：见到一张卡 → 读它的固定 dataMode → inline 直接渲染 data；deferred 先发 data.request 给后端 API /api/account/card/{cardId} 再渲染。

五、API 契约（deferred 模式）

  • 路径：POST /api/account/card/{cardId}（如 /api/account/card/acc03）。
  • 入参：data.request 原样透传，至少含 account_type / card_id / view_mode / dimension / period 等。
  • 出参：与 inline 卡同构的 data 对象。
  • 失败处理：HTTP 非 2xx 或超时 → 该卡整卡隐藏，不展示"加载失败"占位（遵循文档跳过规则）。
